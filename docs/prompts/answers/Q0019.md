# Q0019: 각 계산기별 키패드 기능 논의

**날짜**: 2026-03-03
**원본**: [PROPMTS.md](../PROPMTS.md#q0019)

## 질문

1. 현재 구현된 항목들 기본계산기, 환율 변환기, 단위 변환기, 부가세 계산기의 키패드 버튼 구조에 대한 논의가 필요해
2. 기본 계산기를 제외한 다른 계산기들에서 사직연산 기능이 필요한지? 사용자에게 편리함을 제공하는 기능이 맞는지
3. 각각에서 '-' 버튼이 연산 기능 외에 음수 입력으로의 기능이 필요한지
4. '-' 가 음수 표시로의 기능이 필요가 없다면 '+/-' 버튼도 의미가 없는게 아닌지

## 답변

### 2. 기본 계산기 외 다른 계산기에서 사칙연산이 필요한가?

**결론: 환율·부가세 계산기에서 사칙연산은 실용적이다.**

실제 사용 시나리오:

| 계산기 | 시나리오 | 입력 예시 |
|--------|---------|----------|
| **환율** | 해외여행 중 여러 물건 합산 금액 변환 | `25+30+15` (달러) → 원화 |
| **환율** | 예산에서 차감 후 변환 | `1000-350` (달러) → 원화 |
| **부가세** | 여러 항목 합산 후 세금 계산 | `50000+30000+20000` → 부가세 |
| **부가세** | 할인 적용 후 세금 계산 | `100000-20000` → 부가세 |

사용자가 따로 기본 계산기에서 계산 → 결과를 기억 → 환율/부가세 계산기에서 재입력하는 번거로움을 없앨 수 있다.

**단, `%` 버튼의 필요성은 재검토 필요:**
- 환율 계산기: `500달러의 20%` 같은 사용이 가능하나 빈도가 낮음
- 부가세 계산기: 세율 입력이 별도로 있어 `%`의 역할이 모호함

### 3. 각 계산기에서 `-` 의 음수 입력 기능이 필요한가?

현재 환율 계산기에서는 입력이 `"0"`일 때 `-`를 누르면 음수 모드(`"-"`)로 진입하여 `-300` 같은 음수 금액 입력이 가능하다.

**계산기별 음수 입력 필요성:**

| 계산기 | 음수 필요 | 이유 |
|--------|:---------:|------|
| **기본 계산기** | **O** | 수학 계산에서 음수는 기본 |
| **환율 계산기** | **X** | 음수 금액을 환율 변환하는 실제 시나리오 없음 |
| **부가세 계산기** | **X** | 음수 금액에 부가세를 계산하는 시나리오 없음 |
| **단위 변환기** | **△** | **온도만 필요** (-10°C 등), 나머지 9개 카테고리는 불필요 |

단위 변환기 10개 카테고리 상세:

| 카테고리 | 음수 필요 | 이유 |
|----------|:---------:|------|
| 길이 | X | 음수 거리 불가 |
| 질량 | X | 음수 무게 불가 |
| **온도** | **O** | -10°C, -40°F 실제 존재 |
| 넓이 | X | 음수 면적 불가 |
| 시간 | X | 음수 시간 비실용적 |
| 부피 | X | 음수 부피 불가 |
| 속도 | X | 일반 사용에서 불필요 |
| 연비 | X | 음수 연비 불가 + 변환 오류 위험 |
| 데이터 | X | 음수 데이터 크기 불가 |
| 압력 | X | 절대 압력 기준 음수 불가 |

**권장 조치:**
- 환율 계산기: `input == '0' && key == '-'` 음수 진입 특수 분기 제거
- 부가세 계산기: 동일하게 음수 진입 차단
- 단위 변환기: `+/-` 버튼을 **온도 카테고리에서만** 활성화하거나, 현재처럼 전체 허용 유지 (큰 문제는 아님)

### 4. `-` 음수 기능이 불필요하면 `+/-` 버튼도 의미 없는가?

**맞다. 음수 입력이 불필요한 계산기에서는 `+/-`도 불필요하다.**

현재 상태:
- **기본 계산기**: `+/-` 있음 → **유지** (수학 계산에 필요)
- **환율 계산기**: `+/-` 없음 → 현상 유지 (다만 `-`의 음수 진입 분기는 제거 권장)
- **단위 변환기**: `+/-` 있음 → **온도에서만 필요**, 다른 카테고리에서는 의미 없음
- **부가세 계산기**: `+/-` 없음 → 현상 유지

**단위 변환기 `+/-` 처리 옵션:**

| 방안 | 장점 | 단점 |
|------|------|------|
| A. 온도에서만 `+/-` 표시 | 정확한 UX | 카테고리마다 키패드 레이아웃 변동 |
| B. 전체 유지하되 온도 외 무시 | 레이아웃 일관성 | 불필요한 버튼이 보임 |
| C. `+/-` 제거, 온도는 `-` 직접 입력 | 키패드 단순화 | 음수 입력 UX가 덜 직관적 |

### 종합 권장 사항

| 항목 | 기본 | 환율 | 단위 | 부가세 |
|------|------|------|------|--------|
| 사칙연산 | 유지 | **유지** | 없음 유지 | **유지** |
| % | 유지 | 재검토 | 없음 유지 | 재검토 |
| +/- | 유지 | 없음 유지 | 온도만 검토 | 없음 유지 |
| 음수 진입 (`-`) | 유지 | **제거** | 온도만 | **제거** |
| AC/C 동적 전환 | 유지 | 유지 | AC만 유지 | 유지 |

### 참고: AC/C 동적 전환에 대하여

Apple이 iOS 18에서 C 버튼을 제거했다가 iOS 26에서 다시 복원한 사례가 있다. C 버튼은 백스페이스(⌫)와 기능이 중복되지 않으며, 세 가지 서로 다른 사용자 의도를 커버한다:

| 버튼 | 의도 | 동작 |
|------|------|------|
| ⌫ | "마지막 글자 잘못 쳤어" | 마지막 1자 삭제 |
| C | "이 숫자 자체를 잘못 입력했어" | 현재 피연산자만 삭제, 계산 체인 유지 |
| AC | "처음부터 다시 할래" | 전체 초기화 |

따라서 AC/C 동적 전환은 유지하는 것이 권장된다.
